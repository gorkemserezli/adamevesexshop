## Deferred features (Task 11 starters)

### Per-category page price-range deep-link harness coverage
Current per-category page hydrates `?type=`, `?instock=`, `?price=` from the URL
on load (the `parseFilterFromQuery` path runs and `applyFilter` runs against
the SSR card list). The Task 11 harness covers `?type=` deep-link explicitly
but doesn't cover `?price=` deep-link. The implementation is in place; the
verification gap is the only thing missing.

**Gate:** any reported case of price-deep-link not working in production, or a
content-quality pass that adds price tiers worth deep-linking to.

### Categories description authoring pass
Schema in `src/content/config.ts` accepts `description.optional()` per category.
Today none of the six sample categories ship with a populated description. The
EditorialHero rendering branch on per-category pages only fires when a
description exists. Authoring is a content task, not engineering — but it'd
make the per-category surface visibly richer.

**Gate:** hotel content team writes per-category copy for at least 3 of the 6
categories. Then we ship.

### PDP "Related products" rail (other products in same category)
Natural next step now that the categories surface exists and `category_id` is
indexed in the search records. Implementation: SSR-render up to 4 sibling
products at the bottom of each PDP, reuse `<ProductCard>`. Would require a new
spec section (`§3.21 Related products rail`) since the current §3.8 PDP
anatomy doesn't mention it.

**Gate:** real-device test reveals guests want lateral browsing on PDP, OR
the hotel team requests it explicitly.

### Search filter persistence across page reloads (localStorage)
Currently filter state is URL-only. Some users might expect localStorage
memory of their last filter so reopening the search page restores it.

**Gate:** at least one production support ticket or hotel-team observation
that guests are losing filter state in a way that frustrates them. Until then,
URL-only is the correct behavior — sharable, stateless, no PII concerns.

## Cold-build cost watch (raised by user, Task 11 acceptance)

**Item:** monitor Cloudflare Pages cold-build time as catalog scales. Recorded
baseline at v1.0 ship in README (target documented there). Sharp image
processing dominates cold-build cost.

**Risk forecast:**
- 65 pages → ~6s cold (current measured baseline).
- Catalog scaling to 50+ products × 4 locales = 200+ PDPs would push this further.
- Cloudflare Pages free tier has 500 build minutes/month. Not a current concern,
  but worth tracking as deploy frequency picks up.

**Gate (hard):** cold build exceeds 60s, OR Cloudflare build minutes consumption
trends above 50/month.

**Gate (soft):** content team starts authoring multi-resolution images per
product. At that point sharp invocations multiply and the constant-factor cost
on each shifts the curve.

**Mitigations to consider when the gate trips:**
- Pre-compute image variants and commit them to the repo (skip sharp at build).
- Switch from `sharp` to a lighter-weight image validator if optimization moves
  to a CDN.
- Migrate image hosting to Cloudflare Images, drop sharp from the build path
  entirely.

## Tighten harness network-error allowlist when real stock API ships

The final-pass harness (`scripts/screenshot-task12.mjs`) and Task 7 harness
(`scripts/screenshot-task7.mjs`) currently filter network errors with a
broad allowlist:

```js
text.includes("ERR_CERT") ||
text.includes("ERR_NAME_NOT_RESOLVED") ||
text.includes("Failed to load resource") ||
text.includes(".invalid")
```

This filter is **intentional v1.0-specific** — it suppresses the cert /
DNS errors that the deliberately-unreachable
`https://stock.adamevehotels.invalid/api/stock` host produces on every
page load. With the .invalid host, every PDP fetch fails by design, and
those failures are not bugs.

**Risk if not tightened.** Once `PUBLIC_STOCK_API_URL` swaps to a real
production endpoint, the same allowlist will silently swallow genuine
production failures: real CDN cert problems, real DNS misconfiguration,
real "Failed to load resource" warnings on ANY asset (image 404,
font 403, etc.). The harness reports green; production guests see broken
pages.

**Gate (hard):** `PUBLIC_STOCK_API_URL` is changed away from
`https://stock.adamevehotels.invalid/api/stock`.

**What to do at that gate.**
1. Narrow the allowlist to the EXACT pattern of the real stock API's
   transient errors (if any). Most likely the new allowlist is empty —
   the real API should serve 200s, not cert failures.
2. Verify by running `node scripts/screenshot-task12.mjs` against the
   new endpoint; expect zero filter matches.
3. If the real API has known transient classes (rate limiting, etc.),
   add narrow filters for ONLY those classes, with comments naming the
   exact symptom each filter targets.

**Files to edit.** `scripts/screenshot-task7.mjs` (Task 7 harness has its
own filter inline) and `scripts/screenshot-task12.mjs`. Search for
`.invalid` to locate.
